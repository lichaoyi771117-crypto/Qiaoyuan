## 一、审核总评

**首选方案（MinerU + FinanceToolkit + LLM + Streamlit）在技术架构上完全成立，但财务侧有 4 个必须修正的关键点。如果不改，做出来的系统在国内中小企业场景会翻车。**

---

## 二、逐项财务专业审核

### 2.1 FinanceToolkit — 指标计算引擎 ✅ 但有暗坑

**优点确认**：
- 150+ 指标覆盖面够，PRD 7大类20+指标中95%以上可直接计算
- MIT 协议干净，商用无忧
- 支持本地 DataFrame 输入，不强制依赖其数据获取层

**财务专业发现的坑**：

| 坑点 | 具体表现 | 影响 | 修正建议 |
|------|---------|------|---------|
| **科目映射逻辑** | 默认按 IFRS/US GAAP 科目名称映射（如"Revenue"、"Cost of Goods Sold"），中国准则下为"营业收入"、"营业成本" | 直接对接 MinerU 输出会大量匹配失败 | 必须在科目映射层做中→英翻译，并处理中国准则特有科目（如"税金及附加"在 US GAAP 下无直接对应） |
| **EBITDA 计算口径** | 国外常用 `EBIT + Depreciation + Amortization`，但中国利润表里折旧摊销经常藏在"管理费用""销售费用"里，不单独列示 | 中小企业财报很可能算不出 EBITDA | 预设替代方案：用 `利润总额 + 财务费用` 作为 EBIT 近似，EBITDA 标注为"估算值" |
| **Altman Z-score 原始公式** | FinanceToolkit 自带的 Z-score 是原始美国制造业公式，参数未经中国学者修正 | 直接套用对国内企业误判率高 | 采用 fin-ratios 的版本，或使用中国学者（如张玲等）修正的 Z-score 参数，后续做成可配置项 |

**审核结论**：FinanceToolkit 可用，但必须在上层包装一个"中国适配器"，做科目翻译、公式修正和缺失值兜底。非大改，预计 2-3 天。

---

### 2.2 MinerU — PDF 解析 ✅ 大方向对，需补盲区

MinerU 58k Stars 的中文文档提取能力是开源最强，但财务场景有两个特殊痛点：

**痛点 1：扫描件 + 盖章 + 歪斜的财报**
- 很多中小企业提供的财报是盖章扫描件，甚至手机拍照
- MinerU 对纯电子 PDF 效果好，低质量扫描件会明显下降
- **建议**：在下游加一个质量检测节点，对低置信度页面自动切换 PaddleOCR / 百度 OCR API

**痛点 2：跨页合并单元格**
- 资产负债表动辄 50+ 行，经常跨页
- "资产总计""负债合计"等汇总行容易与明细行混淆
- **建议**：科目映射层增加"汇总行识别规则"——科目名称含"合计""总计"的行，自动标记为校验锚点而非明细科目

---

### 2.3 FinGPT — LLM 报告生成 ⚠️ 需验证

- 训练数据以公开金融新闻、研报为主，**对中小企业非标财报的诊断能力**未经验证
- 很可能出现"说得头头是道，但脱离实际"的幻觉

**建议**：
- 原型阶段直接用 **OpenAI API**（GPT-4o 或 Claude）快速验证效果
- 生产环境要降成本时，再切换到 FinGPT 或 Qwen3 本地部署
- **关键设计原则**：所有数值在计算引擎中确定，LLM 只负责解读，严禁 LLM 自行推导或引用数据

---

### 2.4 AKShare 做行业基准 ⚠️ 方向对，但不完美

AKShare 提供的是 **A 股上市公司**行业均值，而客户是 **中小企业**。两者财务结构差异显著：

- 上市公司的负债率、周转率、ROE 与中小企业不在一个量级
- 用上市公司均值对标中小企业，会导致几乎所有客户都"异常"

**建议**：
- 初期先用 AKShare 行业数据，但报告中明确标注"对标数据来源为 A 股同行业上市公司均值，仅供参考"
- 中长期必须积累自有客户数据库，建立中小企业基准
- 或接入国家统计局《企业绩效评价标准值》（有行业+规模分层，更适合中小企业对标）

---

### 2.5 Gitee 项目判断准确 ✅

Claude Code 准确识别了"上市公司数据派 vs 客户上传派"的根本分歧。ZVT、qstock、stock-monitor 等虽然中文适配好，但场景完全不同——假设数据已通过 API 获取，而非从上传 PDF 中提取。

**延伸价值**：ZVT 的科目体系和 AKShare 的数据源，可作为科目映射层和行业基准的参考资料，不必完全丢弃。

## 架构新增层：数据流与设计详述

从 MinerU 解析完成的结构化数据，到 FinanceToolkit 可计算的标准输入之间，需要经过两个关键新增层的处理。数据流向如下：

```mermaid
flowchart TD
    A[MinerU 解析后的<br/>结构化 DataFrame] --> B
    subgraph B[财务数据质量校验层]
        B1[勾稽关系校验]
        B2[表间勾稽校验]
        B3[科目异常值检测]
        B4[科目逻辑一致性]
        B5[极值合理性检测]
    end
    B -->|校验通过/带警告| C
    subgraph C[中国会计准则适配层]
        C1[科目中→英映射]
        C2[中国特有科目兜底]
        C3[EBITDA 估算替代]
        C4[指标公式修正]
    end
    C --> D[FinanceToolkit<br/>指标计算引擎]```

### 3.1 财务数据质量校验层设计

**目的**：在进入指标计算前，自动发现客户上传财报的基础数据错误，避免"垃圾进垃圾出"。

**校验规则集**：

| 校验项 | 规则 | 处理方式 | 严重级别 |
|--------|------|---------|:--------:|
| **资产负债表平衡** | 资产总计 = 负债合计 + 所有者权益合计 | 不平衡则在报告中标红警告，差值>5%中止分析 | 🔴 阻断 |
| **利润表与资产负债表勾稽** | 期末未分配利润 - 期初未分配利润 ≈ 净利润 - 利润分配 | 差异>10%提示"可能存在未入账利润分配或前期差错更正" | 🟡 警告 |
| **现金流量表校验（如有）** | 期末现金及等价物 = 资产负债表货币资金期末数 | 不一致则提示"现金流量表与资产负债表货币资金科目不匹配" | 🟡 警告 |
| **科目异常值检测** | 货币资金 < 0 / 营业收入 = 0 但净利润 ≠ 0 / 存货 < 0 | 触发时直接标记为"数据异常"，建议客户核对原始报表 | 🔴 阻断 |
| **科目逻辑一致性** | 应收账款 < 0（实为预收账款）/ 应付账款 < 0（实为预付账款） | 自动识别并建议重分类，分析时可自动调整 | 🟡 警告 |
| **极值合理性检测** | 某项资产占比 > 95% / 毛利率 > 95% / 资产负债率 > 100% | 标记为异常，LLM 生成提示时让客户确认是否属实 | 🟡 警告 |

**实现方式**：
- 纯规则引擎，Pandas 向量化判断，不依赖 LLM
- 校验结果以结构化 JSON 输出，包含：`{check_name, status, detail, severity}`
- 阻断级错误需用户确认后才能继续；警告级错误照常计算但报告中标红

**与 Claude Code 现有方案的对接**：
- 该层插在 MinerU 输出之后、科目映射之前
- 因为有些质量问题是科目名称混乱导致的（如"应收帐款"写成了"应收"），此时需要先做模糊科目匹配再做校验
- 建议实际实现时将"质量校验"与"科目映射"设计为可迭代调用：映射后校验→校验失败→回退调整映射→再校验

### 3.2 中国会计准则适配层设计

**目的**：将 MinerU 提取出的中文非标科目名称，翻译为 FinanceToolkit 可识别的标准化科目，并处理中国准则下的特殊逻辑。

**核心功能模块**：

#### 模块 A：科目映射引擎

| 功能 | 说明 |
|------|------|
| **标准科目字典** | 维护一份《中国会计准则常用科目名称 → FinanceToolkit 内部字段》的映射表，如：`{"货币资金": "cash_and_equivalents", "营业收入": "revenue", "税金及附加": "tax_and_surcharge"}` |
| **同义词/别名库** | 兼容企业实际使用的各种别称，如"应收帐款"→"应收账款"、"主营业务收入"→"营业收入"、"应付工资"→"应付职工薪酬" |
| **模糊匹配兜底** | 对未命中字典的科目，使用编辑距离或 LLM 辅助推断归属类别 |
| **科目层级识别** | 自动识别"其中：""合计""总计"等层级标记，构建科目的父子关系，避免将"流动资产合计"当作独立科目计算 |

#### 模块 B：中国特有科目处理

| 特有科目 | 处理逻辑 |
|---------|---------|
| **税金及附加** | 中国利润表中为单独科目，需映射为独立字段，在计算 EBITDA 时作为费用扣除项。IFRS/US GAAP 无直接对应，FinanceToolkit 需扩展。 |
| **营业外收入/支出** | 用于计算"经常性损益"和利润质量判断。需单独标记，不计入毛利率和 EBITDA 计算。 |
| **其他收益** | 政府补助等，中国准则下单独列示，需在盈利能力分析中特别说明（因其不可持续）。 |
| **资产减值损失/信用减值损失** | 中国准则下分别列示，反映资产质量和坏账风险，需纳入 Z-score 修正计算。 |

#### 模块 C：EBITDA 估算替代逻辑
IF 利润表中存在"折旧"和"摊销"单独科目:
EBITDA = 利润总额 + 财务费用 + 折旧 + 摊销
ELSE IF 利润表中存在"资产减值损失"等:
EBITDA ≈ 利润总额 + 财务费用 + 资产减值损失 + 折旧摊销估算值
ELSE:
EBITDA ≈ 利润总额 + 财务费用
并在报告中标注"EBITDA 为近似估算值，因企业未单独披露折旧摊销"

#### 模块 D：指标公式修正配置

| 指标 | 原公式（FinanceToolkit 默认） | 修正后公式（适配中国准则） |
|------|---------------------------|-------------------------|
| 毛利率 | (Revenue - COGS) / Revenue | (营业收入 - 营业成本 - 税金及附加) / 营业收入 |
| EBITDA 率 | EBITDA / Revenue | EBITDA 估算值 / 营业收入 |
| Altman Z-score | 原始美国制造业公式 | 可配置参数：系数1.2/1.4/3.3/0.6/1.0 替换为中国学者修正值 |

**实现方式**：
- 科目映射字典以 YAML/JSON 配置文件维护，方便非技术人员调整
- 映射逻辑封装为 Python 类，对外暴露 `map_to_standard(df)` 方法
- EBITDA 估算逻辑作为策略模式，可切换精确/近似两种策略

两个新增层建议在 Phase 1 加入，预计增加 2-3 天。

## 四、税务模块 — 需额外设计

Claude Code 方案在税务指标上着墨不够。PRD 明确要求**实际税率**和**综合税负率**，但这只是计算的开始。

### 4.1 税务风险检测规则（建议 Phase 3 加入）

| 检测项 | 规则 | 风险等级 |
|--------|------|:--------:|
| 毛利率异常低 | 毛利率 < 行业均值 × 0.5 | ⚠️ 可能被质疑转移定价 |
| 费用率激增 | 管理费用/营收同比增幅 > 50% | ⚠️ 可能虚列费用 |
| 税负率突变 | 实际税率同比变动 > 10个百分点 | 🔴 高风险信号 |
| 关联交易迹象 | 其他应收/应付款占比 > 总资产30% | ⚠️ 可能关联资金占用 |
| 营业外收支过大 | 营业外收支/利润总额 > 20% | ⚠️ 利润质量差，可能调节利润 |

这些规则作为**硬编码规则引擎**，在 LLM 生成报告前先跑一遍，检测到的高风险项直接注入 Prompt，让 LLM 在"税务风险提示"模块重点说明。

### 4.2 融资建议与税务的交叉点

很多中小企业税务申报时压低利润（少交税），但贷款时想展示高利润。系统如果能识别"报表利润"与"纳税申报利润"的差异（后期若可获取数据），可给出"融资前利润修复建议"。此功能属后续迭代（PRD 第 9 节），但架构上先预留数据接口。
