## 一、项目目标

霖信莯咨询面向中小企业客户提供财务分析服务。传统方式下，一份专业财务分析报告需要人工翻阅报表、计算指标、撰写解读，耗时数小时。F-Analyzer 的目标是：

1. **客户上传财务报表（Excel），系统一键生成专业分析报告**，覆盖 7 大类 20+ 项财务指标 + AI 自然语言解读；
2. **所有计算在本地完成**，数据不外传，符合中小企业对数据安全的关切；
3. **报告语言为中文**，面向中小企业主的阅读习惯，避免英文学术术语堆砌；
4. **提供 PDF/HTML 报告导出**，支持客户下载、打印、存档。

> 一句话总结：**把专业财务分析师的"看报表→算指标→写报告"流程自动化，降低中小企业获取专业财务诊断的门槛。**

## 二、技术方案：开源模型 + 中国会计准则适配

### 2.1 总体思路

我们没有选择采购商业财务软件（动辄数万元/年），而是基于开源工具链构建。关键在于：**国际开源财务工具的科目体系和计算公式默认适配 IFRS/US GAAP，直接用于中国企业报表会产生大量错配。** 因此技术路线的核心是"用国际开源做计算引擎 + 自建中国准则适配层"。

### 2.2 技术栈总览

```
[客户上传 Excel/PDF]
       ↓
  自研解析引擎（科目识别 + 格式清洗）
       ↓
  中国会计准则适配层（80+ 科目中→英映射）
       ↓
  质量校验层（勾稽平衡 / 异常值 / 逻辑一致性）
       ↓
  指标计算引擎（7大类 20+ 指标）
       ↓
  LLM 报告生成（DeepSeek-chat）
       ↓
  结构化分析报告 → Streamlit 展示 + HTML 导出
```

### 2.3 各层详细说明

| 层 | 使用的开源技术 | 自建部分 | 说明 |
|----|--------------|---------|------|
| **前端** | Streamlit 1.58 | CSS 定制（中文界面、指标卡片、响应式布局） | Python 全栈，无需前后端分离 |
| **Excel 解析** | openpyxl, xlrd | **自研解析引擎**（6 种真实报表格式适配、多 sheet 遍历、千分位逗号兼容、期初期末列自动识别） | 中国中小企业报表格式差异极大（编制单位不同、会计准则不同），必须自研 |
| **PDF 解析** | PyMuPDF (fitz) | **自研双策略解析器**（ROW_NUM/SPLIT_LINE 两种布局模式） | ⚠️ 目前关闭（见 3.3 节） |
| **科目映射** | — | **完全自建** AccountingMapper（80+ 科目，覆盖资产负债表/利润表/现金流量表） | 这是最核心的"中国化适配"工作 |
| **质量校验** | — | **完全自建** QualityChecker（4 项检查） | 资产负债表平衡、异常负值、科目逻辑、极值合理性 |
| **指标计算** | FinanceToolkit 底层公式 | **自建 MetricCalculator 封装层**（EBITDA 三段式替代估算、Altman Z-score 中国学者修正参数） | 中国企业财报通常不单独列示折旧摊销，需估算替代 |
| **LLM 报告** | OpenAI SDK → DeepSeek API | **自建 ReportGenerator**（5 模块模板 + 中文专业 Prompt 工程） | 用 DeepSeek 替代 OpenAI，成本更低、中文更好 |
| **行业基准** | AKShare（A 股行业均值） | 配置化基准参数 | 需注明 A 股上市公司均值与中小企业的结构差异 |

## 三、开发过程与关键决策

### 3.1 开发阶段

| 阶段 | 内容 | 状态 |
|------|------|:--:|
| **Phase 1** | Streamlit 框架 + 科目映射 + 质量校验 | ✅ 完成 |
| **Phase 2** | 指标计算引擎 + AKShare 基准 | ✅ 完成 |
| **Phase 3** | LLM 报告生成（DeepSeek 集成） | ✅ 完成 |
| **Phase 4** | 报告导出（HTML/CSV）、响应式界面、合规声明 | ✅ 完成 |

### 3.2 关键技术决策

#### 决策一：科目映射必须自建

FinanceToolkit 默认适配 IFRS/US GAAP 科目体系（如 "Current Assets"、"EBITDA"），而中国企业会计准则有独特科目（如"税金及附加"、"营业外收支"、"所有者权益"等）。我们自建了 **AccountingMapper** 模块，逐项将 80+ 中文科目映射到标准英文字段，确保指标计算引擎能正确处理中国企业的财务报表。

#### 决策二：EBITDA 需替代估算

中国中小企业财报通常不单独列示折旧和摊销费用，标准 EBITDA 公式无法直接使用。我们采用了**三段式替代逻辑**：
1. 优先从明细科目中提取折旧/摊销
2. 其次从固定资产变动表中推算
3. 兜底使用行业经验系数估算

所有估算标注了假设前提，避免误导。

#### 决策三：Altman Z-score 参数修正

经典 Altman Z-score 模型基于美国制造业企业数据，参数对中国中小企业不完全适用。我们采用了**中国学者修正参数**（配置化），并标注了适用边界。

#### 决策四：LLM 仅解读不推导

为避免大模型"编造数字"，我们设定了硬性规则：**所有数值在计算引擎中确定，LLM 只负责自然语言解读。** Prompt 模板中明确要求"不要编造任何数字，只使用提供的数值"。

#### 决策五：MVP 阶段优先支持 Excel

首批目标客户中超过 60% 能提供 Excel 格式的财务报表，另有多个客户在财务软件（金蝶、用友）中可直接导出 Excel。因此 MVP 阶段以 Excel 为主，PDF 作为备用通道。

### 3.3 为什么当前不开放 PDF 上传通道

PDF 解析是本项目遇到的最棘手的技术挑战。以下是实事求是的复盘：

**尝试过的方案：**

| 方案 | 结果 | 原因 |
|------|:--:|------|
| **PyMuPDF 文本提取 + 自研解析** | ⚠️ 部分可用 | 对格式规范的巨和物业报表可达到 ~90% 准确率，但不同 PDF 生成器（WPS、金蝶、税务局导出）生成的布局差异极大 |
| **MinerU（上海 AI Lab，58k⭐）** | ❌ 放弃 | 需要 PyTorch（约 2.5GB），对当前部署环境负担过重 |
| **LiteParse（LlamaIndex）** | ❌ 否决 | 本质与 PyMuPDF 同质（文本提取+位置信息），且需 Node.js 依赖 |

**核心困难：**

1. **布局多样性**：中国企业的 PDF 财务报表至少存在 2 种根本不同的布局模式（有行次号型 vs 无行次双栏型），每种模式内部又有细分变体
2. **数字-科目配对**：资产负债表左右双栏的数字在同一行交错排列，且有些 PDF 生成器把数字放在科目名上方而非下方
3. **多页混排**：同一 PDF 中资产负债表、利润表、现金流量表连排，页面间的表格边界需要自动识别
4. **负数表示**：部分 PDF 用会计惯例括号 `(12,345)` 表示负数，正则表达式会漏掉

**当前决策**：保留全部 PDF 解析代码（`src/pdf_parser.py`，约 400 行双策略引擎），等收集更多样本、改进通用适配后再重新开启。

### 4.1 已实现功能

| 模块 | 功能 | 说明 |
|------|------|------|
| **文件上传** | Excel (.xlsx/.xls) | 最大 10MB，多 sheet 自动遍历 |

| **LLM 报告** | 5 模块结构化报告 | 财务数据摘要 · 异常诊断 · 优化建议 · 税务风险提示 · 免责声明 |

### 4.2 当前指标覆盖

**偿债能力**：流动比率、速动比率、资产负债率、利息保障倍数  
**盈利能力**：毛利率、净利率、ROE、ROA、EBITDA 率  
**营运效率**：存货周转天数、应收账款周转天数、总资产周转率  
**成长能力**：营收增长率、净利润增长率、资本积累率  
**现金流**：经营现金流/净利润、自由现金流  
**税务指标**：实际税率、综合税负率  
**财务预警**：Altman Z-score（中国修正参数版）

## 五、后续规划

### 短期（近期可完成）

1. **PDF 解析改进**：收集更多客户 PDF 样本，优化 ROW_NUM 模式负债端提取 + 多页混排页类型自动识别后重新开放
2. **指标基准对照**：接入国家统计局《企业绩效评价标准值》作为第二个行业基准来源
3. **费率优化**：部分指标默认值（如利息保障倍数阈值）需根据行业细分调整

### 中期（需额外调研）

1. **多期对比**：支持上传同期/上期两份报表，自动生成同比分析
2. **图表增强**：在报告中嵌入趋势图、雷达图、瀑布图
3. **报告模板**：支持客户自定义报告模块选择（如仅需融资能力评估）

### 长期（平台化方向）

1. **多用户支持**：数据库持久化，客户历史报告管理
2. **API 服务化**：为第三方系统提供分析接口
3. **行业知识库**：积累各行业典型指标分布，做行业对标
