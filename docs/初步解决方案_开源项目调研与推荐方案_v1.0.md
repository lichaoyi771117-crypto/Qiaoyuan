## 4. 场景适配性深度分析

### 4.1 PRD 核心场景

```
客户（中小企业主/财务人员）
  → 上传 PDF / Excel / 图片格式的财务报表
  → 选择分析目标（财务健康检查 / 融资能力评估 / 降本增效）
  → 系统自动识别报表类型、提取科目、计算指标
  → LLM 生成带解读的分析报告
  → 在线阅读 / 下载 PDF
```

### 4.2 关键需求与开源项目的匹配度矩阵

| PRD 需求 | 关键难点 | 最适合的开源方案 |
|----------|---------|----------------|
| **PDF/Excel/图片上传解析** | 中文财报表格复杂、跨页、合并单元格 | **MinerU**（58k⭐，中文文档提取最强） |
| **中国会计准则科目映射** | "货币资金"→标准字段，需兼容常见科目变体 | **必须自建**，可参考 ZVT / AKShare 的科目体系 |
| **20+核心指标计算** | 7大类指标，含Altman Z-score | **FinanceToolkit**（150+指标）或 **fin-ratios**（含Altman Z） |
| **勾稽关系校验** | 资产=负债+权益，不平标红 | **需自建**，规则引擎实现 |
| **LLM自然语言报告** | 中文财务解读，异常诊断，优化建议 | **FinGPT**（中文金融LLM）或 **OpenAI API**（快速原型） |
| **行业基准对比** | 中国行业均值数据 | **AKShare**（A股行业数据作为参考基准） |
| **Web端交付** | 简洁、响应式、在线/PDF导出 | **Streamlit** 自建（参考 stock-monitor 的UI设计） |
| **税务风险提示** | 中国税制特有（增值税、所得税等） | **需自建**，规则+LLM结合 |

### 4.3 "上市公司数据派" vs "客户上传派" 的根本分歧

| 维度 | Gitee项目（stock-monitor / ZVT / qstock） | PRD 真正需要 |
|------|-------------------------------------------|-------------|
| 数据来源 | Tushare / 东方财富 API（自动获取上市公司数据） | 客户手动上传的 PDF/Excel/图片 |
| 目标用户 | 股民 / 量化投资者 | 中小企业主 / 咨询顾问 |
| 分析对象 | 上市公司（有公开财报） | 非上市公司或 SMEs（无公开数据） |
| 指标体系 | 选股导向（PE/PB/ROE等） | 诊断导向（偿债/盈利/营运/现金流/税务/预警） |
| 输出形式 | 选股信号 / 交易策略 | 专业分析报告 / 优化建议 |

**结论**：Gitee 项目虽然有极好的中文适配，但**场景不匹配**。它们默认用户查询的是已有公开数据的上市公司，而 PRD 要求的是客户上传私有财报文件进行分析。这意味着：

- Gitee 项目可作为**行业基准数据源**的参考
- 但整套系统**不能直接基于 Gitee 项目搭建**，需要以 GitHub 的通用工具链为主

## 5. 最终推荐方案

### 🥇 首选方案：MinerU + FinanceToolkit + LLM + Streamlit（推荐度最高）

#### 架构图

```
┌──────────────────────────────────────────────────────────┐
│                    Streamlit Web 前端                      │
│  [上传] → [选择分析目标] → [处理中...] → [报告展示/下载]     │
└──────────────────────┬───────────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────────┐
│                     PDF/Excel/图片                         │
│                          ↓                                │
│              ┌─────────────────────┐                      │
│              │     MinerU          │  ← 58k⭐ 中文最强     │
│              │  PDF→结构化DataFrame │                      │
│              └─────────────────────┘                      │
│                          ↓                                │
│              ┌─────────────────────┐                      │
│              │  科目映射层(自建)     │  ← 参考中国会计准则    │
│              │ "货币资金"→"cash"    │  + ZVT/AKShare科目    │
│              └─────────────────────┘                      │
│                          ↓                                │
│              ┌─────────────────────┐                      │
│              │  FinanceToolkit     │  ← 150+指标 MIT协议   │
│              │  7大类20+指标计算    │                       │
│              └─────────────────────┘                      │
│                          ↓                                │
│              ┌─────────────────────┐                      │
│              │  勾稽关系校验(规则)   │  ← 自建规则引擎       │
│              └─────────────────────┘                      │
│                          ↓                                │
│              ┌─────────────────────┐                      │
│              │  LLM 报告生成        │  ← FinGPT/OpenAI     │
│              │  异常诊断+优化建议    │                      │
│              └─────────────────────┘                      │
│                          ↓                                │
│              最终交付：专业分析报告                           │
└──────────────────────────────────────────────────────────┘
```

#### 各组件详情

| 层级 | 选型 | 协议 | 理由 |
|------|------|:----:|------|
| **PDF/图片解析** | **MinerU** | ✅ 自定义(Apache基础) | 58k⭐，中文表格提取精度超GPT-4o，支持复杂排版/合并单元格/跨页表格，消费级显卡可跑 |
| **指标计算引擎** | **FinanceToolkit** | ✅ MIT | 150+指标，公式透明可审计，社区活跃(3.5k⭐)，v2.0持续更新。覆盖PRD全部7大类指标 |
| **评分模型** | **fin-ratios**（补充） | ✅ MIT | 含 Altman Z-score、Piotroski F-Score 等10种评分模型，PRD明确要求的破产预警可直接使用 |
| **LLM报告生成** | **FinGPT**（生产）或 **OpenAI API**（原型） | ✅ MIT | FinGPT的中文金融能力+本地部署；原型阶段用OpenAI快速验证 |
| **行业基准** | **AKShare** | ✅ MIT | 9k⭐，免费获取A股行业均值数据，作为对比基准 |
| **科目映射** | **自建** | — | 参考中国《企业会计准则》+ ZVT/AKShare科目体系，预计2-3天 |
| **Web前端** | **Streamlit** 自建 | ✅ Apache | 快速开发，生态成熟，组件丰富 |

#### 与PRD需求对照

| PRD 功能模块 | 实现方式 | 预估工时 |
|-------------|---------|:--------:|
| 4.1 报表输入与识别 | MinerU 解析 + 自建科目映射 + 勾稽校验规则 | 5天 |
| 4.2 财务指标计算 | FinanceToolkit (150+指标) + fin-ratios (Altman Z等) | 3天 |
| 4.3 分析报告生成 | LLM(FinGPT/OpenAI) + 模板化Prompt | 4天 |
| 4.4 网页端交付 | Streamlit 前端（上传→分析→报告→PDF导出） | 5天 |
| 多期对比（9.后延） | FinanceToolkit 原生支持时间序列 | 2天（可后延） |
| 合计 | | **~19天** |

#### 优势

1. **各层松耦合** — 每一层都可独立升级/替换（如后续从 OpenAI 切换到本地 FinGPT）
2. **组件均为各自领域最强** — MinerU(文档解析) + FinanceToolkit(指标计算) 都是业界标杆
3. **科目映射是必经之路** — 无论选哪个方案都需要自建，方案一给了最大灵活度
4. **架构与PRD完全对齐** — 输入→计算→LLM→报告的管线与PRD 4.1~4.4一一对应

#### 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| MinerU 对扫描件PDF的识别准确率 | 先测试5-10份真实客户财报，准确率不达标则叠加PaddleOCR |
| 中小企业财报格式不标准 | 科目映射层内置模糊匹配+同义词表，LLM辅助识别异常科目 |
| FinanceToolkit 默认API依赖（FMP） | 直接使用本地DataFrame输入，不依赖其数据获取层 |
| LLM 生成内容幻觉 | 所有指标数值必须在计算引擎中确定，LLM仅负责解读，并附免责声明 |

### 🥈 备选方案：stock-monitor(Gitee) + 改造

**适用场景**：如果后期发现客户群体中大部分为"已有上市/挂牌公司"，或需要极速出原型

**改造路径**：
1. 保留现有 FastAPI 后端和 Bootstrap 前端框架
2. 增加 PDF 上传和 MinerU 解析模块
3. 替换 Tushare 数据获取为文件输入
4. 扩展指标体系（增加 Altman Z-score、税务指标等）
5. 接入 LLM 生成报告

**工作量**：中高（3-4周），改造幅度较大，不建议作为起步方案

---

### 🥉 备选方案：ZVT + 自研Web

**适用场景**：如果客户场景从"SME咨询"转向"上市公司财务分析"

**优势**：ZVT 已内置中国会计准则下的完整科目体系和 A 股 20+ 年财务数据，可作为行业基准使用

**局限性**：同样是面向上市公司/量化场景设计，SME 场景需较大改造

## 6. 建议启动路径

```
第1周 ─── 第2周 ─── 第3周 ─── 第4周
├─────────┬─────────┬─────────┬─────────┐
Phase 1    Phase 2    Phase 3    Phase 4
│           │           │           │
▼           ▼           ▼           ▼
基础设施   核心管线    LLM集成    打磨交付
                                     │
┌─────────────────────────────────────┘
▼
MVP 可演示
```

### Phase 1：基础设施（Day 1-5）
- 初始化 Streamlit 项目结构
- 集成 MinerU，测试 5-10 份真实中文财报 PDF 的解析效果
- 设计科目映射表（中国《企业会计准则》+ AKShare/ZVT 参考）
- 实现勾稽关系校验规则引擎

### Phase 2：核心管线（Day 6-12）
- 集成 FinanceToolkit，实现 7 大类 20+ 指标计算
- 增加 fin-ratios 的 Altman Z-score 和评分模型
- 建立行业基准数据（AKShare 获取行业均值）
- 前端上传→解析→计算的端到端流程打通

### Phase 3：LLM 集成（Day 13-17）
- 设计报告生成 Prompt 模板（财务摘要→异常诊断→优化建议→税务风险）
- 接入 LLM（原型阶段建议 OpenAI API，后续切换 FinGPT）
- 实现报告模块生成和展示
- 增加数据异常的标红和警告

### Phase 4：打磨交付（Day 18-21）
- PDF 报告导出（ReportLab / WeasyPrint）
- 响应式设计适配移动端
- 免责与合规声明
- 用户体验优化
