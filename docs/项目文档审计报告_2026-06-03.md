## 一、审计结论

**整体判定：✅ 方案为目前可确认的最优方案，选型逻辑已形成闭环，审核闭环完成，可以进入 MVP 开发阶段。**

四个支撑理由：

1. **选型逻辑闭环** — MinerU（中文文档提取最强）→ FinanceToolkit（指标计算最成熟）→ LLM（最灵活），每层均为业界标杆且协议干净
2. **审核闭环完成** — 财务审核报告 V1.1 的 4 个修正点（科目映射 / EBITDA 替代 / Z-score 参数修正 / 税务规则）均已吸收到 CLAUDE.md 和解决方案中
3. **风险可控** — 所有已识别风险（扫描件质量 / 科目不规范 / LLM 幻觉 / 行业基准偏差）均有明确缓解措施
4. **可执行性强** — 4 个 Phase 共约 19 天，MVP 阶段优先支持 Excel，有明确验证节点

## 三、需补充的缺口

以下三个缺口中，前两项应在进入 Phase 1 开发前明确，第三项可在 Phase 1 数据库设计时预留。

### 缺口 1：税务指标计算公式确认

| 项目 | 说明 |
|------|------|
| **缺口描述** | PRD 4.2 要求计算**实际税率**（所得税费用/利润总额）和**综合税负率**（各项税费支出/营收），但 FinanceToolkit 默认按 IFRS/US GAAP 设计，是否自带这两个中国特有税务指标**需要验证** |
| **影响范围** | Phase 2 指标计算管线 |
| **建议处理方式** | 搭最小 Demo 时，用 FinanceToolkit 跑一遍税务指标。如果缺，在科目映射层之后增加**税务指标计算模块**（公式简单，纯代数运算，不需要额外依赖） |
| **预估工时** | 不自带则 +0.5 天 |

### 缺口 2：指标雷达图可视化方案

| 项目 | 说明 |
|------|------|
| **缺口描述** | PRD 4.3 要求"指标雷达图与行业对标"展示，目前方案中未指定具体可视化技术选型 |
| **影响范围** | Phase 3 报告展示模块 |
| **选项 A** | **Plotly**（交互式，Streamlit 原生支持，雷达图 + 悬停数据标签，开源免费） |
| **选项 B** | **ECharts**（Apache 协议，国内生态好，更美观，通过 streamlit-echarts 集成） |
| **建议** | 推荐 **Plotly**，原因：与 Streamlit 集成最顺滑（st.plotly_chart），零额外配置，交互功能满足需求 |
| **预估工时** | 2 天（含行业基准对比线绘制） |

### 缺口 3：多期对比的时间序列存储预留

| 项目 | 说明 |
|------|------|
| **缺口描述** | PRD 第 9 节要求后续支持"多期报表对比分析（趋势图）"，虽标为后延，但 Phase 1 数据库设计时不预留时间序列字段会导致后续需要数据迁移 |
| **影响范围** | Phase 1 数据库 Schema 设计 |
| **建议处理方式** | 在 SQLite 的财务报表表中增加 `report_period`（报告期）和 `report_date`（报表日期）字段，使同一企业可存储多期数据。后续对比分析时只需按日期 GROUP BY |
| **预估工时** | 几乎没有额外工时，只是设计表结构时多两个字段 |
